Avoid deadlocks!

Let's talk about deadlocks and some ways of avoiding them

March 26, 2022

  • blog
  • dev
  • deadlocks
  • async

What is a deadlock?

A deadlock, in computer science, refers to a specific condition when two or more processes are each waiting for another to release a resource the processes share. In other works, both processes are asking the question, “Hey, can you let me know when you’re done with that? so I can continue”, which clearly, is not being answered for any of them and hence the deadlock.

Example of a deadlock

Example Environment

We will be explaining this example in C# for .NET Framework and .NET Core. Find .NET Core 6 source code repo here

Example Objective

We have a list of postcodes and we want to verify if the postcodes do really exist. For that, we will send a POST request to an API’s endpoint to verify them. Our postodes will be stored in a List<string> called postcodesList, which will be the used as the parameter for the following action.

Code

sub-action

public static async Task<PostcodesResponse> ArePostcodesValid(List<string> postcodesList)
{
    /*Creating JSON Object containing the postcodes*/
    var postcodesData = new { postcodes = postcodesList.ToArray() };
    /*Parsing JSON Object to string*/
    string jsonPostcodesData = JsonConvert.SerializeObject(postcodesData);
    /* 
    Encapsulating json encoded in UTF8 
    format into content for POST HTTP Request 
    */
    var content = new System.Net.Http.StringContent(jsonPostcodesData, Encoding.UTF8, "application/json");
    var httpClient = HttpClientNoHeaderAccessor.HttpClient;
    var response = await httpClient.PostAsync(BASEURL, content);
    if (response.IsSuccessStatusCode)
    {
        string dataResponse = await response.Content.ReadAsStringAsync();
        /*Parsing string text reponse into PostcodesResponse Entity*/
        return JsonConvert.DeserializeObject<PostcodesResponse>(dataResponse);
    }
    else
    {
        return null;
    }
}

Code Flow

First of all, let’s keep in mind that main-action refers to ValidatePostcodes() and sub-action refers to ArePostcodesValid(postcodesList).

  1. main-action calls sub-action.
  2. Within the same context and thread started in main-action, sub-action starts a HTTP POST request by executing PostAsync() (which in this case, is awaited, which means that a background thread, different than main one from main-action, will be executing this task.).
  3. PostAsync() first returns an uncompleted task, indicating that the TTP POST request is not completed yet.
  4. Therefore, sub-action awaits the task returned by PostAsync(). At this moment, the context is captured and it will be used to continue executing sub-action when the POST request is completed.
  5. After this, sub-action returns an uncompleted task to the top level method.
  6. Top level method, due to its .Result function, will be now synchronously waiting until sub-action is completed - This will block the context on the main thread as .Result is a synchronous wait (opposite of await).
  7. Eventually, the REST request will complete by the Thread Pool (a.k.a background threads), which will complete the task returned by PostAsync().
  8. The continuation of the sub-action (code after line var response = await httpClient.PostAsync(BASEURL, content);) is ready to proceed, although it needs to have something crutial for it to continue its execution, the context, (which was previously captured in step 4) so it can continue its execution.
  9. Deadlock happens - main-action is currently blocking the captured context waiting for the sub-action to complete. sub-action is waiting for the context to be free so it can complete its execution on main thread.

How can we fix this?

There might be a few out there to avoid this deadlock but I’ll go through 2 different ones, choosing between one or another depends on how you have your project setup.

Using ConfigureAwait(ContinueOnCapturedContext:false)

This solution will suit if you prefer to stick with synchronous code rather than asynchronous one.

Keeping in mind steps above, a few things change now:

  • Within step 4, due to specifying ContinueOnCapturedContext:false, context won’t be captured at this point and this task will be given to the Thread Pool to continue.
  • Within step 8, once PostAsync() completes, the Thread Pool won’t need the main context anymore as it was specified to do so and will continue its execution through sub-action - This will then return its Task Result to the main-action, which was synchronously waiting for it. Code flow will continue.

Using ASYNC/AWAIT operations

This solution will suit if you rather use async code flow, which in my opinion, is the best alternative due to the benefits async provides - I will go through this later on.

We will change main-action to

PostcodesResponse postcodesResponse = 
    await CustomClasses.PostcodeAPIProcessor.ArePostcodesValid(postcodesToCheck);
  • Now, main thread will execute the main-action and will await for ArePostcodesValid(postcodesToCheck), which means that current context will be captured and thread pool will execute this function in the background - The best part of this is that, while the background threads are working on that task, main thread is asynchronously waiting for it to end (async) - The fact that the main thread is asyncrhonously waiting, means that it will be used where is needed in other async code blocks (it won’t be synchronously waiting without doing anything and wasting resources).
  • Within sub-action, Thread pool will also await the function PostAsync(), its context will be captured and the task will also be handled to Thread pool to execute.
  • Eventually, once PostAsync() returns with a result, the captured context will be used to continue sub-action’s code returning result from sub-action to main-action.
  • Main thread with its original context will be pinged with a signal that their execution through rest of main-action is ready to go - Main thread with its original context will come back to it and finish it.
